# 🏥 AI智能问诊系统 - 运行指南

## 项目简介

**AI Medical Consultant**（人工智能医疗问诊系统）是一个基于 **LLM大模型 + Agent架构 + RAG检索增强** 的智能医疗问诊系统。

| 项目 | 说明 |
|------|------|
| 技术栈 | Vue3 + FastAPI + LangChain + FAISS + PostgreSQL |
| LLM模型 | DeepSeek Chat（兼容OpenAI接口） |
| Embedding | 通义千问 text-embedding-v1 |
| 向量数据库 | FAISS |
| 核心能力 | 智能问诊、知识库检索、分诊决策、鉴别诊断、流式对话 |

## 快速开始（本地开发）

### 第一步：安装后端依赖

```bash
cd backend
pip install -r requirements.txt
```

> **注意**：如果在 Windows 上安装 `faiss-cpu` 遇到问题，可以尝试：
> ```bash
> pip install faiss-cpu
> # 或者跳过FAISS，仅使用ChromaDB（代码会自动降级）
> ```

### 第二步：配置环境变量

后端目录中已经包含 `.env` 文件，预配置了：

- ✅ **LLM**: DeepSeek Chat API（已填入你的 API Key）
- ✅ **Embedding**: 通义千问 text-embedding-v1（已填入你的 API Key）
- ⚠️ **数据库**: 默认使用 PostgreSQL（需要在下一步选择数据库方案）

如果需要修改配置，编辑 `backend/.env` 文件。

### 第三步：选择数据库方案

#### 方案A：使用 SQLite（最简单，无需安装数据库）

修改 `backend/.env` 中的数据库配置：

```env
DATABASE_URL=sqlite+aiosqlite:///./data/medical.db
DATABASE_URL_SYNC=sqlite:///./data/medical.db
```

并在 `requirements.txt` 中添加：
```
aiosqlite==0.20.0
```

同时修改 `backend/app/models/database.py`，将数据库引擎改为 SQLite。

#### 方案B：使用 PostgreSQL + Redis（生产环境）

使用 Docker 启动 PostgreSQL 和 Redis：

```bash
docker compose up -d postgres redis
```

### 第四步：初始化医学知识库

```bash
cd backend
python -m data.medical_kb.init_kb
```

此脚本会导入示例医学知识（感冒、高血压、糖尿病、药品说明等）到 RAG 向量数据库。

预期输出：
```
============================================================
初始化医学知识库...
============================================================
✓ 已导入 7 条医学知识文档
  知识库总文档数: 7
  索引向量数: 7
============================================================
```

### 第五步：启动后端服务

```bash
cd backend
python -m uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload
```

或者双击运行 `start_backend.bat`。

启动成功后：
- **API 文档**: http://localhost:8000/docs
- **健康检查**: http://localhost:8000/health

### 第六步：启动前端（可选）

如果需要 Web 界面：

```bash
cd frontend
npm install
npm run dev
```

访问 http://localhost:5173

## 使用流程

### 1. 用户注册/登录

```
POST /api/v1/user/register
POST /api/v1/user/login
```

通过前端界面注册账号并登录，或者在 Swagger 文档中测试。

### 2. 创建问诊会话

```
POST /api/v1/consult/
```

每次问诊需要先创建一个会话。

### 3. 开始智能问诊

使用 WebSocket 连接进行实时流式对话：

```
ws://localhost:8000/api/ws/chat?token=<your-jwt-token>
```

发送消息格式：
```json
{
  "consultation_id": 1,
  "message": "我头疼两天了，还有点发烧，怎么办？"
}
```

Agent 处理流程：
1. **意图识别** → 判定为"问诊类"
2. **症状分析** → 提取"头痛、发烧、持续两天"
3. **追问生成** → 可能追问"有没有其他症状？""体温多少？"
4. **RAG检索** → 从医学知识库检索相关知识
5. **LLM生成** → 流式生成回复
6. **分诊建议** → 推荐就诊科室和紧急程度

### 4. 查看问诊记录

```
GET /api/v1/consult/
GET /api/v1/consult/{id}
```

### 5. 知识库检索

```
POST /api/v1/knowledge/search
{
  "query": "高血压的症状",
  "top_k": 5
}
```

### 6. 添加医学知识

```
POST /api/v1/knowledge/documents
{
  "title": "流行性感冒",
  "category": "disease",
  "content": "流行性感冒是由流感病毒引起的..."
}
```

或上传文档：
```
POST /api/v1/knowledge/upload
Content-Type: multipart/form-data
file: 医学知识.pdf
category: disease
```

## 常见问题

### 1. LLM API 调用失败

**症状**：返回"AI服务暂时不可用"

**解决**：
- 检查 `LLM_API_KEY` 是否正确
- 测试 API 连通性：
  ```bash
  curl https://api.deepseek.com/v1/models -H "Authorization: Bearer sk-e63595f598c34bf3a8affebbbc0be260"
  ```

### 2. Embedding 模型加载失败

**症状**：`sentence_transformers` 报错或 DashScope API 超时

**解决**：
- 检查 `EMBEDDING_API_KEY` 配置
- 如果是网络问题，可以本地下载 embedding 模型
- 代码会自动降级到 DashScope API

### 3. PostgreSQL 连接失败

**症状**：`Can't connect to PostgreSQL`

**解决**：
- 检查 Docker 是否启动：`docker compose ps`
- 或切换到 SQLite（见第三步方案A）

### 4. FAISS 安装问题（Windows）

**症状**：`pip install faiss-cpu` 失败

**解决**：
```bash
# 使用 conda
conda install -c conda-forge faiss-cpu

# 或下载预编译wheel
pip install https://github.com/kyamagu/faiss-wheels/releases/download/v1.7.3/faiss_cpu-1.7.3-cp311-cp311-win_amd64.whl
```

### 5. WebSocket 连接问题

**症状**：前端无法连接 WebSocket

**解决**：
- 检查后端是否启动在 8000 端口
- 检查 `CORS_ORIGINS` 配置是否包含前端地址
- 检查 Vite 代理配置

## 模型配置说明

| 配置项 | 值 | 说明 |
|--------|-----|------|
| LLM_API_KEY | `sk-e63595f...` | DeepSeek API Key |
| LLM_BASE_URL | `https://api.deepseek.com` | DeepSeek API地址 |
| LLM_MODEL | `deepseek-chat` | 使用的模型 |
| EMBEDDING_API_KEY | `sk-ws-H.RYDXILX...` | 千问 Embedding API Key |
| EMBEDDING_MODEL | `text-embedding-v1` | 千问 Embedding 模型 |

> ⚠️ 如需使用其他模型（如 GPT-4），只需修改 `LLM_BASE_URL` 和 `LLM_MODEL` 即可。

## 架构说明

### Agent 决策流程

```
用户输入 → 意图识别 → ═══ 分支处理 ═══
                      ├─ 问诊类: 症状分析 → 追问/收集 → RAG检索 → LLM诊断
                      ├─ 查询类: RAG检索 → LLM回答
                      └─ 闲聊类: LLM直接回复
                                    ↓
                              流式输出到前端
                                    ↓
                              持久化存储消息
```

### RAG 检索流程

```
用户Query → Query Expansion（问题改写）
              ├─ 同义词替换
              ├─ 症状标准化
              └─ 问句补全
                  ↓
          ┌───────┼───────┐
          ↓       ↓       ↓
      稠密检索  稀疏检索  术语匹配
      (FAISS)  (BM25)  (关键词)
          └───────┼───────┘
                  ↓
           RRF融合排序
                  ↓
            上下文组装
          (System Prompt + 知识 + 历史)
                  ↓
              LLM生成
```
